## 🧠 ЧАСТЬ 1: Как работает событие под капотом

Когда ты кликаешь на кнопку, браузер не просто вызывает твой код. Он запускает сложный процесс из трёх фаз. Понимание этого спасает от багов вида «почему клик сработал дважды?» или «почему родитель перехватил клик ребёнка?».

### 1.1 Три фазы события (Event Flow)
Представь матрёшку: `<html>` → `<body>` → `<div>` → `<button>`. Ты кликнул на `<button>`.

1.  **Фаза погружения (Capturing):**
    *   Браузер идёт **сверху вниз** от `window` до целевого элемента.
    *   По пути он проверяет, есть ли слушатели с `{ capture: true }`.
    *   *Используется редко* (в основном фреймворками или для глобальных перехватчиков).

2.  **Целевая фаза (Target):**
    *   Событие достигло элемента, на который кликнули (`<button>`).
    *   Выполняются все слушатели, повешенные на этот элемент.

3.  **Фаза всплытия (Bubbling):**
    *   Браузер идёт **снизу вверх** от цели к `window`.
    *   На каждом уровне проверяет, есть ли обычные слушатели (`addEventListener('click', ...)`).
    *   *Это самая важная фаза.* Именно она позволяет использовать делегирование.

📌 **Важно:** Не все события всплывают! `focus`, `blur`, `mouseenter`, `mouseleave` — **не всплывают**. `click`, `keydown`, `input`, `submit` — **всплывают**.

📖 *Спецификация:* [UI Events – Event order](https://w3c.github.io/uievents/#event-flow)

## 🧠 ЧАСТЬ 2: Объект Event и управление потоком

Первый аргумент функции-обработчика — это объект `Event`. Он содержит всю информацию о том, что произошло.

### 2.1 Ключевые свойства
| Свойство | Описание | Пример использования |
|----------|----------|----------------------|
| `target` | Элемент, на котором произошло событие (самый глубокий) | Кликнули по `<span>` внутри кнопки → `target` будет `<span>` |
| `currentTarget` | Элемент, на котором **висит** текущий обработчик | В делегировании: `currentTarget` — это список, `target` — кнопка внутри |
| `type` | Тип события | `"click"`, `"keydown"` |
| `key` / `code` | Какая клавиша нажата (для keyboard events) | `event.key === 'Enter'` |

### 2.2 Методы управления
| Метод | Что делает | Когда использовать |
|-------|------------|-------------------|
| `stopPropagation()` | Останавливает всплытие/погружение дальше по дереву | Редко. Только если нужно изолировать компонент (например, модалка внутри модалки) |
| `preventDefault()` | Отменяет действие браузера по умолчанию | Форма не отправляется, ссылка не переходит, чекбокс не меняется |

⚠️ **Антипаттерн:** `return false` в jQuery или старых скриптах. В современном `addEventListener` это **не работает** как `preventDefault`. Используй явный метод.

## 🧠 ЧАСТЬ 3: Делегирование событий (Event Delegation)

Это **главный паттерн** работы с динамическим DOM.

### 3.1 Проблема подхода «на каждый элемент»
```js
// ❌ Плохо: 100 кнопок = 100 слушателей в памяти
document.querySelectorAll('.btn').forEach(btn => {
  btn.addEventListener('click', handler);
});

// ❌ Проблема: Если добавить новую кнопку через JS, на ней НЕ будет обработчика.
// Придётся вешать его заново.
```

### 3.2 Решение: Делегирование
Мы вешаем **один** слушатель на общего родителя (контейнер), а внутри проверяем, на кого именно кликнули.

```js
const list = document.getElementById('todo-list');

list.addEventListener('click', (event) => {
  // 1. Ищем ближайшего предка с нужным классом/атрибутом
  const deleteBtn = event.target.closest('.delete-btn');
  
  // 2. Если кликнули не туда — выходим (guard clause)
  if (!deleteBtn) return;

  // 3. Выполняем логику
  const li = deleteBtn.closest('li');
  li.remove();
});
```

✅ **Преимущества:**
1.  **Производительность:** 1 слушатель вместо 1000. Меньше расход памяти.
2.  **Динамика:** Новые элементы, добавленные в DOM позже, автоматически обрабатываются. Не нужно перевешивать события.
3.  **Чистота кода:** Логика группы элементов собрана в одном месте.

🧠 **Ментальная модель:**
Делегирование — это «почтовое отделение». Письма (события) приходят в одно место (родитель), а сортировщик (`closest`) решает, кому их доставить.

## 🧠 ЧАСТЬ 4: Опции addEventListener

Третий аргумент `addEventListener` — это не только `true/false` (capture), но и объект настроек.

```js
element.addEventListener('click', handler, {
  once: true,      // Сработает 1 раз и сам удалится (идеально для тултипов, модалок)
  passive: true,   // Сообщает браузеру: "я не буду делать preventDefault". 
                   // Ускоряет скролл на тач-устройствах (обязательно для touchstart/wheel)
  capture: false   // Слушать на фазе всплытия (по умолчанию)
});
```

📌 **Правило:** Всегда используй `{ passive: true }` для событий скролла (`wheel`, `touchstart`, `touchmove`), если не планируешь блокировать скролл. Это требование Chrome и стандарт производительности.

## 🧩 МЕНТАЛЬНЫЕ МОДЕЛИ И ПРАВИЛА

1.  **События всплывают.** Используй это для делегирования.
2.  **`target` vs `currentTarget`:** `target` — кто инициировал, `currentTarget` — кто слушает. В делегировании они почти всегда разные.
3.  **Делегирование — дефолт для списков.** Никогда не вешай слушатели на элементы списка в цикле, если их может быть много или они динамические.
4.  **`preventDefault` отменяет браузер, `stopPropagation` отменяет родителей.** Не путай.
5.  **Отписка важна.** Если компонент удаляется из DOM, удаляй и его специфичные слушатели (если они не делегированы), чтобы избежать утечек памяти.

3.  В `script.js` реализуй:
    *   **Добавление задачи:**
        *   По клику на `#add-btn` (или Enter в инпуте) создавай `<li>` с структурой:
            ```html
            <li data-id="timestamp">
              <span>Текст задачи</span>
              <div class="actions">
                <button class="btn-edit">✏️</button>
                <button class="btn-delete">🗑️</button>
              </div>
            </li>
            ```
        *   Используй `Date.now()` для `data-id`.
        *   Очищай инпут после добавления.
    *   **Делегирование на `#task-list`:**
        *   Повесь **один** `click` listener на `<ul>`.
        *   Внутри проверяй `event.target.closest()`:
            *   Если клик по `.btn-delete` → удаляй `<li>`.
            *   Если клик по `.btn-edit` → заменяй текст `<span>` на `<input>` для редактирования. При потере фокуса (`blur`) или Enter сохраняй текст обратно в `<span>`.
            *   Если клик по самому `<li>` (но не по кнопкам) → переключай класс `completed` (зачёркивание).
    *   **Защита от пустых задач:** Не добавляй задачу, если инпут пустой или содержит только пробелы (`trim()`).
    *   **Комментарии:** Объясни в коде, почему мы используем делегирование, а не вешаем события на каждую кнопку при создании.
