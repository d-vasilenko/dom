## DOM-дерево, подключение JS, DevTools

## 🧠 ЧАСТЬ 1: Как браузер превращает HTML в DOM (под капотом)

### 1.1 Путь от байтов к дереву
Когда браузер получает HTML-документ, он не «читает» его как текст. Он запускает **HTML-парсер**, который работает строго по спецификации и выполняет следующие шаги:

1. **Декодирование** → байты превращаются в символы (с учётом `<meta charset>`).
2. **Токенизация** → поток символов разбивается на токены: `<startTag>`, `<endTag>`, `text`, `comment`, `doctype`.
3. **Построение дерева** → токены преобразуются в узлы (`Node`). Формируется иерархия: `Document → Element → Text/Comment`.
4. **DOM готов** → дерево доступно через глобальный объект `document`.

📌 **Важно:** парсер работает **синхронно в главном потоке**. Любая блокировка этого потока (тяжёлый JS, синхронные запросы, `document.write`) останавливает построение DOM и замораживает интерфейс.

### 1.2 Узлы DOM и их типы
В спецификации DOM каждый узел реализует интерфейс `Node`. У него есть строгие свойства:

| Свойство | Описание | Пример для `<p id="msg">Текст</p>` |
|----------|----------|-----------------------------------|
| `nodeType` | Числовой тип узла | `1` (ELEMENT_NODE) |
| `nodeName` / `tagName` | Имя узла (для элементов всегда ВЕРХНИЙ РЕГИСТР) | `"P"` |
| `textContent` | Сырой текст всех дочерних текстовых узлов | `"Текст"` |
| `childNodes` | Живая коллекция всех дочерних узлов (включая текстовые и комментарии) | `NodeList [Text, Comment]` |

🔢 **Константы `nodeType`:**
- `1` → `ELEMENT_NODE` (`<div>`, `<p>`, `<input>`)
- `3` → `TEXT_NODE` (пробелы, переносы, текст между тегами)
- `8` → `COMMENT_NODE` (`<!-- -->`)
- `9` → `DOCUMENT_NODE` (`document`)

💡 **Ментальная модель:** DOM ≠ исходный HTML. DOM – это **живое объектное дерево**, которое браузер постоянно синхронизирует с рендерингом. Изменил узел → браузер пересчитал layout/paint (если изменение затрагивает геометрию или стили).

📖 *Спецификация:* [DOM Standard – Node interfaces](https://dom.spec.whatwg.org/#interface-node)  
📖 *Парсинг HTML:* [HTML Living Standard – Tree construction](https://html.spec.whatwg.org/multipage/parsing.html#tree-construction)

## ⚡ ЧАСТЬ 2: JavaScript и DOM: порядок загрузки и выполнения

### 2.1 Как браузер встречает `<script>`
Парсер идёт сверху вниз. Когда он видит `<script>`, поведение зависит от атрибутов:

| Вариант | Загрузка | Выполнение | Влияние на парсинг | Когда использовать |
|---------|----------|------------|-------------------|-------------------|
| `<script src="app.js">` в `<head>` | Блокирующая | Сразу после загрузки | ❌ Останавливает парсинг HTML | ❌ Почти никогда |
| `<script src="app.js">` в конце `<body>` | Блокирующая | После парсинга DOM | ✅ DOM уже существует, но рендер ждёт загрузки | ✅ Работает, но устарело |
| `<script src="app.js" defer>` | Параллельная | После полного парсинга DOM, **до** `DOMContentLoaded` | ✅ Не блокирует, порядок скриптов сохраняется | ✅ **Рекомендуется** |
| `<script src="app.js" async>` | Параллельная | Сразу как загрузится, **прерывает парсинг** | ⚠️ Порядок не гарантируется | ✅ Только для независимых скриптов (аналитика, виджеты) |
| `<script type="module" src="app.js">` | Параллельная | Как `defer` + strict mode + CORS | ✅ Современный стандарт | ✅ Для ES-модулей |

### 2.2 Почему `defer` – стандарт де-факто
- Скрипт загружается **параллельно** с парсингом HTML → страница не ждёт сеть.
- Выполняется **строго после** построения DOM → `document.getElementById` никогда не вернёт `null` из-за порядка.
- Порядок нескольких `defer`-скриптов **сохраняется** → предсказуемая инициализация.
- Выполняется **до** события `DOMContentLoaded` → можно безопасно вешать слушатели на это событие или работать с DOM сразу.

🧠 **Ментальная модель:**  
`HTML-парсер` и `JS-движок` делят один главный поток. `defer` развязывает сеть и выполнение, сохраняя детерминированность. Это единственный атрибут, который даёт **производительность + предсказуемость**.

📖 *Спецификация:* [Script processing model – defer/async](https://html.spec.whatwg.org/multipage/scripting.html#attr-script-defer)

## 📝 ЧАСТЬ 3: Чтение и изменение узлов: `textContent` vs `innerHTML` vs `innerText`

### 3.1 Три способа работы с содержимым
| Метод | Что делает | Парсит HTML? | Безопасность | Производительность | Когда использовать |
|-------|------------|--------------|--------------|-------------------|-------------------|
| `textContent` | Читает/записывает сырой текст всех дочерних текстовых узлов | ❌ Нет | ✅ XSS невозможен | ⚡ Быстро, нет парсинга | ✅ 95% случаев: данные, сообщения, UI-текст |
| `innerHTML` | Читает/записывает HTML-разметку дочерних узлов | ✅ Да | ⚠️ XSS при непроверенных данных | 🐌 Медленнее, перестраивает дерево | ✅ Только для доверенной/статичной разметки |
| `innerText` | Читает/записывает **видимый** текст (учитывает CSS `display:none`, `visibility`) | ❌ Нет | ✅ Безопасно | 🐌 Вызывает reflow, нестандартное происхождение | ⚠️ Только если нужно именно видимое представление |

### 3.2 Почему `innerText` – ловушка
- Изначально проприетарное свойство IE, позже стандартизировано с оговорками.
- При чтении **вызывает пересчёт layout**, чтобы определить, какие узлы видимы.
- При записи ведёт себя непредсказуемо с переносами строк и скрытыми элементами.
- В современном коде **избегай**, если нет явной задачи «получить только то, что видит пользователь».

### 3.3 Правило безопасности и производительности
```js
// ❌ Антипаттерн (XSS + медленный парсинг)
el.innerHTML = userInput;

// ✅ Безопасно и быстро
el.textContent = userInput;

// ✅ Если нужна разметка – создавай узлы явно
const span = document.createElement('span');
span.textContent = userInput;
el.append(span);

// ✅ Если чужой HTML неизбежен – санитайз
import DOMPurify from 'dompurify';
el.innerHTML = DOMPurify.sanitize(untrustedHTML);
```

📖 *MDN:* [Node.textContent vs HTMLElement.innerText](https://developer.mozilla.org/en-US/docs/Web/API/Node/textContent)

## 🔍 ЧАСТЬ 4: DevTools как профессиональный инструмент отладки DOM

### 4.1 Elements: живое дерево, а не исходник
- Панель Elements показывает **текущее состояние DOM**, а не `view-source`.
- Правый клик по узлу → `Break on →`:
  - `subtree modifications` – пауза при добавлении/удалении дочерних узлов
  - `attribute modifications` – пауза при изменении атрибутов/классов
  - `node removal` – пауза при удалении узла
- Это заменяет сотни `console.log` и позволяет точно увидеть, **какой JS-код мутирует DOM**.

### 4.2 Console: скрытые возможности
| Команда | Что делает |
|---------|------------|
| `$0` | Последний выбранный элемент в Elements |
| `$$('selector')` | Аналог `document.querySelectorAll` |
| `console.dir(el)` | Показывает JS-объект узла (свойства, прототипы), а не DOM-дерево |
| `monitorEvents(el, 'click')` | Логирует все события указанного типа на элементе |
| `getEventListeners(el)` | Возвращает зарегистрированные слушатели (Chrome) |

### 4.3 Network & Sources
- **Network → JS**: видно порядок загрузки, статус `defer`/`async`, кэширование, размер.
- **Sources → Breakpoints**: ставь точки останова не только на строки, но и на DOM-мутации (через Elements). Смотри Call Stack, чтобы понять цепочку вызовов.
- **Performance**: позже будем использовать для замера reflow/repaint, но уже сейчас привыкай открывать эту вкладку при подозрении на «тормоза».

## 🧩 МЕНТАЛЬНЫЕ МОДЕЛИ И ПРАВИЛА (запомни)

1. **DOM ≠ HTML.** DOM – живое объектное дерево. Исходник – только стартовая точка.
2. **Главный поток один.** Парсинг HTML, выполнение JS, расчёт layout, отрисовка – всё в одном потоке. Блокируешь JS → блокируешь UI.
3. `defer` – **дефолт** для скриптов приложения. `async` – только для независимых метрик/виджетов.
4. **Текст → `textContent`. Разметка → `createElement` + `append`. Чужой HTML → санитайз.**
5. **DevTools – микроскоп.** DOM-брекпоинты и `console.dir` экономят часы отладки.
6. **`nodeType === 1`** – элемент. Всё остальное – текст, комментарии, документ. Путаница здесь ломает логику обхода дерева.
