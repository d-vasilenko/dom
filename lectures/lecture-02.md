## 🧠 ЧАСТЬ 1: Методы поиска элементов

| Метод | Возвращает | Область поиска | Селекторы | Примечание |
|-------|------------|----------------|-----------|------------|
| `getElementById(id)` | `Element` или `null` | Только `document` | Только `id` | Самый быстрый. Не работает на произвольных узлах. |
| `getElementsByClassName(class)` | `HTMLCollection` (живая) | `document` или любой `Element` | Только классы | Возвращает коллекцию, обновляется при мутациях DOM. |
| `getElementsByTagName(tag)` | `HTMLCollection` (живая) | `document` или любой `Element` | Только теги | Аналогично выше. Редко нужен в современном коде. |
| `querySelector(selector)` | Первый `Element` или `null` | `document` или любой `Element` | Любой CSS-селектор | Останавливается на первом совпадении. |
| `querySelectorAll(selector)` | `NodeList` (статический) | `document` или любой `Element` | Любой CSS-селектор | Возвращает снимок DOM на момент вызова. |

📌 **Важно:** `querySelector*` работают чуть медленнее `getElement*` на огромных деревьях, но в 99% задач разница измеряется микросекундами. Выбор метода диктуется **не скоростью, а типом коллекции и удобством селекторов**.

## 🧠 ЧАСТЬ 2: Живые (Live) vs Статические (Static) коллекции

Это **ключевая концепция** модуля. Ошибка здесь ломает циклы, вызывает пропуски элементов или бесконечные итерации.

### 2.1 Как браузер хранит коллекции
- `HTMLCollection` (от `getElementsBy*`) и `NodeList` от `childNodes` → **живые**. Браузер не копирует узлы. Он хранит внутренний указатель на DOM и **пересчитывает коллекцию при каждом обращении**.
- `NodeList` от `querySelectorAll` → **статический**. Это моментальный снимок (snapshot). Браузер копирует ссылки на узлы в момент вызова. Дальнейшие мутации DOM на коллекцию не влияют.

### 2.2 Ловушка итерации по живой коллекции
```js
const items = document.getElementsByClassName('item');
for (let i = 0; i < items.length; i++) {
  items[i].remove(); 
  // ❌ length уменьшается, индексы сдвигаются. 
  // Элементы пропускаются или цикл ведёт себя непредсказуемо.
}
```

✅ **Безопасные паттерны:**
```js
// 1. Снимок в массив (рекомендуется)
const snapshot = [...document.getElementsByClassName('item')];
snapshot.forEach(el => el.remove());

// 2. Итерация с конца (для живых коллекций)
for (let i = items.length - 1; i >= 0; i--) {
  items[i].remove();
}

// 3. Использовать querySelectorAll (статический)
document.querySelectorAll('.item').forEach(el => el.remove());
```

🧠 **Ментальная модель:**  
Живая коллекция = указатель на DOM. Статическая = фотография DOM.  
Если DOM меняется во время обхода → работай с фотографией.

## 🧠 ЧАСТЬ 3: `closest()` и `matches()`

Ручные циклы `while (el = el.parentElement)` ушли в прошлое. Современные методы работают быстрее, безопаснее и читаемее.

### 3.1 `element.matches(cssSelector)`
- Проверяет, соответствует ли элемент селектору.
- Возвращает `true` / `false`.
- **Не ищет**, только проверяет текущий узел.
```js
const btn = document.querySelector('.btn');
btn.matches('[data-action="add"]'); // true/false
```

### 3.2 `element.closest(cssSelector)`
- Идёт **вверх** по дереву, начиная с самого элемента.
- Возвращает первого совпавшего предка или `null`.
- Останавливается на `document` (не уходит за пределы дерева).
```js
const span = document.querySelector('.btn span');
span.closest('.wrapper'); // найдёт .wrapper, даже если клик был глубоко внутри
```

📌 **Почему это важно:** `closest()` – фундамент делегирования событий. Он надёжно находит целевой компонент, даже если пользователь кликнул по иконке, тексту или псевдоэлементу внутри.

## 🧠 ЧАСТЬ 4: Кэширование ссылок и управление памятью

### 4.1 Зачем кэшировать?
Каждый вызов `querySelector*` или `getElementsBy*` заставляет браузер:
- парсить селектор (если не закэширован движком)
- обходить дерево или пересчитывать живую коллекцию
В обработчиках событий, циклах или `requestAnimationFrame` повторный поиск = просадки FPS.

✅ **Паттерн:**
```js
const list = document.getElementById('list'); // один раз
const items = list.querySelectorAll('.item'); // один раз
// дальше работаем с list и items
```

### 4.2 Когда кэширование вредно?
Если DOM динамически перестраивается (элементы удаляются и создаются заново), старая ссылка становится **stale** (указывает на detached node). Такой узел:
- не отображается на странице
- остаётся в памяти, пока жива JS-ссылка → утечка
- методы вроде `closest()` на нём работают, но он не часть рендер-дерева

✅ **Правило:**  
DOM стабилен → кэшируй.  
DOM пересоздаётся → ищи заново или используй делегирование (разберём в модуле событий).

## 🧩 МЕНТАЛЬНЫЕ МОДЕЛИ И ПРАВИЛА
1. `querySelectorAll` → статический снимок. `getElementsBy*` → живая ссылка.
2. Живые коллекции мутируют при изменении DOM → итерация по ним опасна без конвертации в массив.
3. `closest()` идёт вверх (включая себя), `matches()` проверяет текущий узел. Оба принимают CSS-селекторы.
4. Кэшируй ссылки, если DOM стабилен. Не кэшируй, если узлы пересоздаются.
5. `element.querySelector` ищет только внутри элемента, а не по всему документу.
